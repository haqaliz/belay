## 2. Implementation phases

Each phase is one TDD unit: RED ‚Üí GREEN ‚Üí REFACTOR, suite green at the end, one commit.

### Phase 1 ‚Äî The record type and registry I/O

- **Goal:** `InstanceRecord` + strict load/dump, so every later phase has a typed input.
- **Prerequisites:** none.
- **Files:** create `eval/instances/__init__.py`, `eval/instances/registry.py`;
  create `tests/test_eval_instance_registry.py`.
- **Design:**
  - `@dataclass(frozen=True) class InstanceRecord` with fields:
    `instance_id: str`, `repo: str`, `base_commit: str`, `problem_statement: str`,
    `task_string: str`, `is_control: bool = False`.
  - `load_registry(path: Path) -> tuple[InstanceRecord, ...]`
  - `dump_registry(records, path) -> None` ‚Äî stable key order, trailing newline, so a
    regenerated file diffs cleanly.
  - **Fail loud:** a missing/blank required field raises a named `ValueError` naming the
    field and the offending `instance_id`. Never default to `None`/`""`. Mirrors the
    ledger's fail-closed loader (`src/belay/phase0/ledger.py:147-197`) ‚Äî read it first and
    match its error style.
- **Tests (write first):** spec criteria 7 and 8 ‚Äî
  `test_registry_round_trips`, `test_missing_field_is_a_named_error`,
  `test_blank_instance_id_is_rejected`,
  `test_control_instances_are_marked_and_separable` (a `controls()`/`real()` split helper).
- **Validation:** `uv run pytest tests/test_eval_instance_registry.py`
- **Commit:** `eval(instances): InstanceRecord + fail-closed registry I/O`

### Phase 2 ‚Äî Task-string derivation

- **Goal:** deterministic, mechanical task strings from `problem_statement`.
- **Prerequisites:** Phase 1.
- **Files:** create `eval/instances/tasks.py`, `tests/test_eval_instance_tasks.py`.
- **Design:**
  - `derive_task_string(problem_statement: str, *, max_chars: int = 1500) -> str`
  - Pure. No LLM, no network, no clock, no randomness.
  - Normalize whitespace; truncate on a **paragraph/sentence boundary** at or under
    `max_chars` rather than mid-word; prefix a fixed imperative framing (a module-level
    constant, e.g. "Fix the following issue in this repository:").
  - Empty/whitespace-only statement ‚Üí named `ValueError`, never a silently empty task.
  - **Document the coverage caveat in the module docstring:** truncation can omit part of
    the requirement, so the agent attempts what the truncated statement says. This is a
    stated limitation in the results, not a defect.
- **Tests (write first):** spec criterion 6 ‚Äî
  `test_task_string_is_derived_deterministically`,
  `test_task_string_is_bounded_and_non_empty`,
  `test_truncation_falls_on_a_boundary_not_mid_word`,
  `test_empty_problem_statement_raises`.
- **Validation:** `uv run pytest tests/test_eval_instance_tasks.py`
- **Commit:** `eval(instances): deterministic task-string derivation`

### Phase 3 ‚Äî The stratified selection function

- **Goal:** the draw, deterministic and loud on shortfall. **The correctness-critical phase.**
- **Prerequisites:** Phases 1‚Äì2.
- **Files:** create `eval/instances/selection.py`, `tests/test_eval_instance_selection.py`.
- **Design:**
  - Module constants (from the verified dataset survey ‚Äî see PRD ¬ßTechnical):
    ```python
    PURE_PYTHON_REPOS = frozenset({
        "django/django", "sympy/sympy", "pytest-dev/pytest",
        "sphinx-doc/sphinx", "psf/requests", "pylint-dev/pylint", "pallets/flask",
    })
    LARGE_REPOS = ("django/django", "sympy/sympy")   # the top-up pool, order = tie-break
    ```
    Everything else (matplotlib, scikit-learn, astropy, xarray, seaborn) is excluded ‚Äî
    they need a C/Cython/OpenMP toolchain and Debian-only system packages.
  - `select_instances(pool, *, target: int, seed: int) -> tuple[InstanceRecord, ...]`
    1. filter to `PURE_PYTHON_REPOS`;
    2. take **all** small-repo instances (everything not in `LARGE_REPOS`);
    3. if that already meets/exceeds `target`, deterministically sample down to `target`;
    4. else top up from django and sympy, **alternating** so the split stays balanced,
       each repo's candidates ordered by a seeded shuffle;
    5. if the eligible pool is smaller than `target` ‚Üí **raise** `InsufficientPoolError`
       naming pool size and target.
  - Determinism: `random.Random(seed)` only ‚Äî never module-level `random`, never the clock.
    Sort by `instance_id` before any shuffle so input ordering cannot leak in.
- **Tests (write first):** spec criteria 1‚Äì5, on a **synthetic** pool (not the real data,
  so the test doesn't break when the registry is regenerated) ‚Äî
  `test_selection_is_deterministic_given_a_seed`,
  `test_selection_takes_all_small_repo_instances_before_topping_up`,
  `test_selection_balances_the_topup_between_django_and_sympy`,
  `test_selection_excludes_non_pure_python_repos`,
  `test_selection_raises_when_pool_is_smaller_than_target`,
  plus `test_selection_is_insensitive_to_input_ordering`.
- **Validation:** `uv run pytest tests/test_eval_instance_selection.py`
- **Commit:** `eval(instances): deterministic stratified instance selection`

### Phase 4 ‚Äî The regeneration script + committed data

- **Goal:** a re-runnable fetch, and the committed `pool.json` / `selected.json`.
- **Prerequisites:** Phases 1‚Äì3.
- **Files:** create `eval/scripts/fetch_swebench_pool.py`,
  `tests/fixtures/swebench_rows_sample.json`; generate `eval/instances/pool.json` and
  `eval/instances/selected.json`; extend `tests/test_eval_instance_registry.py`.
- **Design:**
  - Stdlib only: `urllib.request`, `json`, `argparse`.
  - Paginate `https://datasets-server.huggingface.co/rows?dataset=princeton-nlp%2FSWE-bench_Lite&config=default&split=test&offset={0,100,200}&length=100`.
  - **Separate the pure logic from the I/O**: `rows_to_pool(rows: list[dict]) ->
    tuple[InstanceRecord, ...]` is pure and tested against the recorded fixture; the
    network call is a thin `main()` the tests never invoke.
  - Apply the strict filters: pure-Python repos, single-file patch, ‚â§15 changed lines.
  - Write `pool.json` via `dump_registry`, then `selected.json` =
    `select_instances(pool, target=<launch target>, seed=<fixed>)`.
  - **Record the seed and target inside `selected.json`** (a small header object) so the
    draw is reproducible and auditable from the committed file alone.
- **Control instances:** append the 2‚Äì3 controls (PRD must-have 15) to `selected.json`
  with `is_control: true`. Their `task_string` is a hand-written trivially-correct edit,
  **not** derived from a `problem_statement`. Flag to the reviewer: control *design* is a
  judgment call ‚Äî the agent should propose them and **stop for human review** rather than
  inventing them silently.
- **Tests (write first):** `test_rows_to_pool_filters_to_strict_constraints`,
  `test_rows_to_pool_is_pure_and_offline` (no network access in the test),
  `test_committed_selected_registry_loads_and_meets_the_target`,
  `test_committed_pool_contains_no_excluded_repos`.
- **Validation:** `uv run pytest tests/test_eval_instance_registry.py` + full suite.
- **Commit:** `eval(instances): stdlib pool fetch + committed stratified mint set`

## 4. Testing strategy

- **All deterministic, all offline, all CI-safe.** No `manual` marker needed in this
  aspect ‚Äî nothing here spends money or needs Darwin.
- Selection tests use a **synthetic** pool so regenerating the real registry never breaks
  them; only the two "committed data" tests read the real files, and they assert
  *properties* (no excluded repos, target met), not exact contents.
- Fetch-script tests read `tests/fixtures/swebench_rows_sample.json` ‚Äî a small recorded
  slice, not the full 300 rows.
- Commands: `uv run pytest tests/test_eval_instance_*.py`, then `uv run pytest` for the
  whole suite before each commit.

## 5. Environment & determinism

- No new env vars, no new dependencies, no external services.
- Determinism requirements, all enforced by tests:
  - selection is a pure function of `(pool, target, seed)`;
  - `random.Random(seed)` only ‚Äî no module-level `random`, no clock, no `Date`-like input;
  - inputs sorted by `instance_id` before shuffling so dict/file ordering cannot leak in;
  - `dump_registry` emits stable key order so regeneration produces a clean diff.

## 6. Edge cases & error handling

| Case | Behavior |
|---|---|
| Eligible pool < target | `InsufficientPoolError` naming pool size + target. **Never a silently short draw** ‚Äî that becomes a short denominator, which is the R6 false-zero failure mode one layer up. |
| Missing/blank registry field | Named `ValueError` with field + `instance_id`. Never `None`-defaulted. |
| Empty `problem_statement` | Named `ValueError`. Never an empty task string. |
| `problem_statement` shorter than `max_chars` | Returned whole; no padding. |
| HF API unreachable | The *script* fails loudly with the URL and status. Tests never hit the network, so this cannot break CI. |
| A repo appears in the data that is in neither the include nor exclude list | Excluded (allow-list semantics, not deny-list) **and** logged by the script, so a dataset change is visible rather than silent. |
| Duplicate `instance_id` in the pool | Named `ValueError` at load. |

**Verdict impact: none.** This aspect emits no verdicts and touches no axis. It cannot
produce `UNVERIFIED`. It only supplies inputs to a later capture.

## 7. Agent execution notes

**Checkpoints for human review:**
1. **After Phase 3** ‚Äî the selection logic is the correctness-critical piece; the
   concentration mitigation (django/sympy balance) is the whole point of the aspect.
2. **Before committing `selected.json`** ‚Äî this fixes which instances the live mint runs,
   and therefore what the published number covers. Human confirms the launch target
   (~65‚Äì70 for a ‚â•50 denominator) and the seed.
3. **Control-instance design (Phase 4)** ‚Äî the agent must **propose and stop**, not invent.
   A badly-designed "clean" control that actually does violate something would void a
   valid mint.

**Likely iteration:** the exact HF API response shape (field names in `rows[].row`) ‚Äî
verify against the recorded fixture before writing the parser.

**Ask before proceeding if:**
- the strict filters yield fewer than the launch target (the survey says 166 eligible, so
  this would mean the filter is wrong ‚Äî do not silently loosen it);
- any change to `src/belay/` seems necessary (it should not be ‚Äî flag instead);
- the dataset's live composition differs materially from the surveyed distribution.

**Spec ambiguities flagged:**
- **Launch target is not yet a fixed number** ‚Äî the PRD says "~65‚Äì70" for a ‚â•50
  denominator. Phase 4 needs one concrete integer; confirm at checkpoint 2.
- **`max_chars = 1500` is an assumption**, not a measured value. It is a judgment call
  balancing task fidelity against prompt size; revisit if Stage-2 traces show the agent
  missing the requirement.
- **Control-instance count** is "2‚Äì3"; pick 3 unless cost argues otherwise (a single
  control failing is ambiguous; two failing is a signal).
