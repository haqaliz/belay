## 0. Ground truth the executing agent must not re-derive

Confirmed by reading the code in this worktree, not from memory:

- **Manifest layout** (`src/belay/sandbox/gate.py:330`):
  `manifest_root = Path(BELAY_SNAPSHOT_DIR).resolve().parent / f"{name}.manifests"`.
  So `BELAY_SNAPSHOT_DIR=<inst>/snapshots` ⇒ manifests at `<inst>/snapshots.manifests/`,
  snapshot trees at `<inst>/snapshots/turn-NNNN/`.
- **Trace name** (`src/belay/trace.py:145-149`): `trace-<UTC-stamp>-<8hex>.jsonl`, generated
  at proxy start — **carries no instance id**.
- **What the stock CLI resolves** (`src/belay/phase0/runner.py:74-83`):
  `default_manifest_dir_for(trace) = trace.parent / (trace.stem + ".manifests")`.
  `belay phase0 run` has **no `--manifest-dir` flag** — this convention is the only path.
- **Tree paths are absolute** (`replay/persist.py:109`), so snapshot trees can stay where
  the gate wrote them; only the trace file and the `.manifests` dir need moving.
- **`run_session`** (`eval/minting_driver/session.py:42`) drives ONE instance, propagates
  every exception, closes the transport once in a `finally`.
- **`gated_env` / `proxy_command`** (`eval/minting_driver/capture.py`) already build the
  env and the `python -m belay.proxy <server...>` argv (no `--` separator). Reuse them.
- **`filesystem_server_command` / `shell_server_command`** (`eval/minting_driver/servers.py`)
  return `["node", "<abs entrypoint>", ...]`. **Never `npx`** — it hangs behind the gate.
- **`run_batch`** (`src/belay/phase0/runner.py`) is the read-only consumer; the harness
  produces its input directory and never modifies `src/belay/`.

### The rename bridge, concretely

Per instance, after `run_session` returns:

```
FROM (as the gated proxy wrote them):
  <inst>/traces/trace-<ts>-<hex>.jsonl
  <inst>/snapshots.manifests/<handle>.json
  <inst>/snapshots/turn-NNNN/...            (trees; absolute paths inside manifests)

TO (what the stock `belay phase0 run <batch-dir>` resolves):
  <batch-dir>/trace-<instance-id>.jsonl
  <batch-dir>/trace-<instance-id>.manifests/<handle>.json
  (trees stay put)
```

Exactly one `trace-*.jsonl` is expected in `<inst>/traces/`. **Zero or more-than-one is an
error, not a pick-the-first** — a mint that produced no trace, or somehow two, is not a
thing to paper over. This guard is the difference between the rename bridge and a
silent fake-PIVOT.

## 2. Implementation phases

Each phase is one TDD unit: RED → GREEN → REFACTOR, suite green, one commit. Ordered so
the correctness-critical bridge is proven before anything depends on it.

### Phase 1 — The rename bridge (`bridge.py`)

- **Goal:** turn one instance's gated capture into the exact layout `belay phase0 run`
  resolves. **The highest-value code in the aspect.**
- **Prereqs:** none (pure filesystem logic).
- **Files:** create `eval/minting_driver/bridge.py`, `tests/test_minting_driver_bridge.py`.
- **Design:**
  - `bridge_capture(*, instance_id, trace_dir, snapshot_dir, batch_dir) -> Path`
    returns the path of the renamed trace.
  - Find exactly one `trace-*.jsonl` in `trace_dir`; **raise a named error** on zero or
    on more than one.
  - Compute the source manifests dir as the gate does:
    `snapshot_dir.resolve().parent / f"{snapshot_dir.resolve().name}.manifests"`.
    Import and reuse this from a single source of truth if one is exposed; otherwise
    replicate it with a comment citing `gate.py:330` so the coupling is visible.
  - Move (`shutil.move`, not copy) trace → `batch_dir/trace-<id>.jsonl` and manifests dir
    → `batch_dir/trace-<id>.manifests/`. Trees are left where they are.
  - Idempotence guard: a destination that already exists is a named error, never a silent
    overwrite (two instances colliding on an id is a bug to surface).
- **Tests (write first):** batch-harness spec criterion 2 —
  `test_bridge_produces_the_layout_phase0_run_resolves` (assert
  `phase0.runner.default_manifest_dir_for(renamed_trace)` points at the moved manifests
  dir, and that dir contains the handle json);
  `test_bridge_raises_when_no_trace_is_present`,
  `test_bridge_raises_when_multiple_traces_are_present`,
  `test_bridge_refuses_to_overwrite_an_existing_destination`,
  `test_bridge_leaves_snapshot_trees_in_place`.
  Build the FROM layout by hand in `tmp_path` (a fake trace file + a fake
  `snapshots.manifests/<handle>.json`) — no proxy, no sandbox, fully offline.
- **Validation:** `uv run pytest tests/test_minting_driver_bridge.py`
- **Commit:** `eval(batch): the rename bridge from gated capture to phase0-run layout`

### Phase 2 — Resume checkpoint (`checkpoint.py`)

- **Goal:** record per-instance status so a crash does not restart from zero.
- **Prereqs:** none.
- **Files:** create `eval/minting_driver/checkpoint.py`,
  `tests/test_minting_driver_checkpoint.py`.
- **Design:**
  - A small JSON ledger: `{instance_id: {status, reason, trace_path}}` where status ∈
    `{"captured", "failed"}`. Fail-closed load mirroring `phase0/ledger.py`.
  - `load_checkpoint(path) -> Checkpoint`, `Checkpoint.record(id, status, ...)`,
    `Checkpoint.is_done(id) -> bool` (both captured and failed count as done for the pass —
    a failed instance is not silently retried forever; re-running is an explicit choice),
    `Checkpoint.save(path)`.
  - Atomic write (temp file + `os.replace`) so a crash mid-save cannot corrupt the ledger.
- **Tests (write first):** `test_checkpoint_round_trips`,
  `test_recording_is_atomic_across_a_simulated_crash` (write to a temp, replace),
  `test_is_done_is_true_for_captured_and_failed`,
  `test_corrupt_checkpoint_is_a_named_error`.
- **Validation:** `uv run pytest tests/test_minting_driver_checkpoint.py`
- **Commit:** `eval(batch): resumable per-instance checkpoint ledger`

### Phase 3 — Workspace prep (`workspace.py`)

- **Goal:** materialize an instance's repo at `base_commit` in the sibling layout the gate
  requires.
- **Prereqs:** `instance-registry` (`InstanceRecord`).
- **Files:** create `eval/minting_driver/workspace.py`,
  `tests/test_minting_driver_workspace.py`.
- **Design:**
  - `prepare_workspace(record, *, root, clones_dir) -> WorkspaceLayout` where
    `WorkspaceLayout` carries `work_dir`, `trace_dir`, `snapshot_dir` with
    **`snapshot_dir` a sibling of `work_dir`, never inside it** (`gate.py:303-315`).
  - Repo acquisition: maintain a **cached bare clone per repo** under a gitignored
    `clones_dir`; create the instance's `work_dir` via `git worktree add --detach
    <work_dir> <base_commit>` (or `git clone --shared` + checkout). A fresh full clone per
    instance is wasteful across 50 django instances — reuse the bare clone.
  - The git calls are I/O; isolate them behind a thin injectable `runner=subprocess.run`
    seam so tests substitute a fake and **never hit the network or run git**.
  - A pure `layout_for(instance_id, root) -> WorkspaceLayout` computes paths and is tested
    directly; `prepare_workspace` composes layout + acquisition.
- **Tests (write first):** batch-harness spec criterion 7 —
  `test_snapshot_dir_is_a_sibling_of_the_scope_never_inside_it`,
  `test_layout_paths_are_deterministic_for_an_instance_id`,
  `test_prepare_invokes_git_with_the_base_commit` (fake runner records argv),
  `test_prepare_is_offline_in_tests` (no real subprocess/network),
  `test_prepare_surfaces_a_git_failure_as_a_named_error`.
- **Validation:** `uv run pytest tests/test_minting_driver_workspace.py`
- **Commit:** `eval(batch): per-instance workspace prep at base_commit (sibling layout)`

### Phase 4 — The sequential batch driver (`batch.py`)

- **Goal:** tie it together — for each instance: prep → drive one gated session → bridge →
  checkpoint — sequentially, one call in flight, fresh client per instance, errors
  contained.
- **Prereqs:** Phases 1–3, the timeout threading, `servers.py`, `instance-registry`.
- **Files:** create `eval/minting_driver/batch.py`, `tests/test_minting_driver_batch.py`.
- **Design:**
  - `run_mint(records, *, root, model_factory, server_command, checkpoint_path,
    request_timeout, max_steps, system) -> Checkpoint`.
  - **`model_factory: Callable[[tools], Model]` — called once per instance** so no
    conversation state leaks between instances (`anthropic_client.py:84-87`). The factory,
    not a shared model, is the seam tests inject.
  - Sequential `for record in records`: skip if `checkpoint.is_done`; else prep workspace,
    build `gated_env` + `proxy_command(server_command)`, `run_session(...,
    request_timeout=...)`, then `bridge_capture(...)`, then `checkpoint.record("captured")`
    and `save`.
  - **Per-instance `try/except Exception`**: record `failed` with the reason and continue
    (mirrors `run_batch`'s `ERRORED` discipline, `runner.py:123-135`). One `ServerExited`
    never aborts the mint.
  - **Two segregated batches**: `server_command` is one server; the caller runs the mint
    twice (filesystem, shell) into two different `batch_dir`s. `run_mint` never mixes
    servers into one trace dir — assert the invariant, don't just document it.
  - Sequential only; no threads (`transport.py:213-215`).
- **Tests (write first):** batch-harness spec criteria 1, 3, 4, 5, 6, 9 —
  `test_drives_instances_sequentially_never_more_than_one_call_in_flight` (re-entrancy
  counter across instances, fake transport),
  `test_a_failing_instance_does_not_abort_the_batch`,
  `test_resume_skips_already_captured_instances`,
  `test_resume_after_partial_instance_is_retried_or_recorded_failed_not_left_partial`,
  `test_each_instance_gets_a_fresh_model_client` (factory call count == instance count),
  `test_request_timeout_is_forwarded_to_run_session`,
  `test_two_servers_never_share_a_trace_dir`.
  All with fakes: fake model factory, fake transport factory, a stub workspace-prep
  (inject the prep function too, so no git runs in CI).
- **Validation:** `uv run pytest tests/test_minting_driver_batch.py`
- **Commit:** `eval(batch): sequential resumable mint driver over the registry`

### Phase 5 — INSTRUMENT SUSPECT end-to-end (darwin-gated)

- **Goal:** prove the false-zero defense on the real harness→`phase0 run`→`report` path:
  a mint with ~no verifiable turns reads as `INSTRUMENT SUSPECT`, not 0%.
- **Prereqs:** Phases 1–4.
- **Files:** create `tests/test_phase0_batch_short_denominator.py`.
- **Design:**
  - Reuse the existing darwin-gated e2e machinery (`tests/test_phase0_e2e.py`). Drive a
    tiny batch whose captures are all trace-only / no-verifiable-turn (e.g. a bridged
    layout with empty manifests), run `phase0.report`, assert the `INSTRUMENT SUSPECT`
    string and the **absence** of a violation-rate line.
  - Marker: platform-gated like the existing e2e; **not** `manual` (no spend, no model —
    it uses fixtures, not a live mint). Card acceptance #2.
- **Tests (write first):**
  `test_short_denominator_batch_reports_instrument_suspect_not_zero_percent`.
- **Validation:** `uv run pytest tests/test_phase0_batch_short_denominator.py`
  (on Darwin; skips elsewhere).
- **Commit:** `eval(batch): short-denominator mint reports INSTRUMENT SUSPECT (R6 defense)`

## 4. Testing strategy

- **Everything in Phases 1–4 is deterministic, offline, CI-safe.** No `manual` marker;
  no model, no git, no npx, no network. The seams (`model_factory`, `transport_factory`,
  the workspace-prep injection, the git `runner`) exist precisely so CI never spends or
  reaches out.
- Phase 5 is **platform-gated** (Darwin/Seatbelt) like `test_phase0_e2e.py`, but uses
  fixtures, not a live mint — so it runs in a macOS CI without spend.
- The actual live mint over real instances is `mint-execution`, out of scope here.
- Commands: `uv run pytest tests/test_minting_driver_batch.py` (etc.), then `uv run
  pytest` whole-suite before each commit, then `uv run ruff check eval tests`.

## 5. Environment & determinism

- New env: none required by the harness itself (it consumes `BELAY_EVAL_SERVER_ROOT` via
  `servers.py`, and sets the three `BELAY_*` gate vars via `gated_env`). Model keys are the
  caller's (`mint-execution`).
- Gitignored dirs the harness writes: repo clones and mint output. Add
  `/eval/clones/` and `/eval/mint/` to `.gitignore` in Phase 3/4 (the executing agent adds
  these; they are the user's run data — no-egress guardrail).
- Determinism required and tested: instance **order** follows the registry (no reordering);
  path layout is a pure function of `instance_id`; nothing here reads the clock or random.
  (The mint's model calls are non-deterministic — that lives above this layer and is a
  stated property, not this aspect's concern.)

## 6. Edge cases & verdict impact

| Case | Behavior |
|---|---|
| No trace produced for an instance | `bridge_capture` raises; `run_mint` records `failed` with the reason; batch continues. **Never** treated as a real short instance. |
| Two traces in one instance's trace dir | Named error — a mint invariant was violated; surface it. |
| Destination id collision in `batch_dir` | Named error — never a silent overwrite. |
| `ServerExited` / `ReplyTimeout` mid-instance | Caught per-instance, recorded `failed`, continue. |
| Crash mid-mint | Re-entry loads the checkpoint and skips done instances; a partially-written instance is retried or recorded failed, never left as a partial trace. |
| git checkout fails (bad base_commit) | Named error from `prepare_workspace`; instance recorded `failed`; **the selection function can draw a replacement** (registry aspect). |
| snapshot_dir inside scope | Prevented by `layout_for`'s sibling placement; asserted by test. |

**Verdict impact: none.** This aspect emits no verdicts and touches no axis. It cannot
produce `UNVERIFIED` itself — it *arranges* captures so `run_batch` (A1+A2) can. The one
verdict-adjacent risk is the bridge silently mis-wiring manifests so `run_batch` returns
all-UNVERIFIED → `INSTRUMENT SUSPECT` (a fake PIVOT). Phase 1's criterion 2 and Phase 5
exist specifically to close that.

## 7. Agent execution notes

**Checkpoints for human review:**
1. **After Phase 1** — the bridge is the fake-PIVOT guard; its criterion-2 test is the one
   that must genuinely assert `default_manifest_dir_for` resolves to real manifests, not
   just that files moved.
2. **After Phase 4** — before `mint-execution` spends anything, the whole harness should be
   green against fakes, and Stage 1 of `mint-execution` (a real single-instance capture)
   is what finally validates the bridge against a real proxy — no test substitutes for it.

**Likely iteration:** the git worktree/clone mechanics in Phase 3 (bare-clone reuse vs
`--shared`); pin the approach against one real repo by hand before wiring the fake runner.

**Ask before proceeding if:**
- any change to `src/belay/` seems needed (it should not — the bridge is the reason);
- the harness starts wanting agent retries-with-reflection, planning, or memory — that is
  agent-framework drift (guardrail #1); retrying a *failed instance* is fine, making the
  *agent* smarter is not.

**Dispatch note for the integrator:** Phases 1, 2, 3 touch disjoint files and can run in
parallel. Phase 4 depends on all three. Phase 5 depends on Phase 4.
