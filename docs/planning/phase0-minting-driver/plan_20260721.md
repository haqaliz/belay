## 0. Ground rules (every task inherits these)

- **Isolation contract (load-bearing):** all new product-side code lives under `eval/`. It imports
  **nothing** from `src/belay` except by spawning `python -m belay.proxy` as a subprocess. It
  reuses `belay.replay.client` internals **only** in test/eval code that already has `belay` on the
  path — but prefer copying the ~30 lines of stdlib send/await into `eval/` to keep `eval/`
  standalone. Decide in Task 2; either way, **`tests/test_import_guard.py` must stay green** and no
  new dependency lands in the `belay` package.
- **No `belay` subcommand.** Nothing in `src/belay/cli.py` changes.
- **Guardrail #1 fence:** one `tools/call` → wait → repeat. No planning, no memory across tasks, no
  multi-step autonomy, no tool authoring. If a task tempts you past that, STOP and flag.
- **BYOK / no egress:** model key from env only; never a vendor default; nothing proxied; no
  raw-state egress. The two real clients are import-isolated so the driver core imports neither SDK.
- **Determinism split:** everything in CI is deterministic and offline (fake model + fake
  transport). Every real-model / real-server / darwin path sits behind an env-guarded manual marker
  and **never runs in CI** (mirrors the C8 model-seam discipline).

## 2. Implementation phases

Tasks 1–4 are the CI-green core (the shippable DoD). Tasks 5–8 are should-haves; 5 and 7 are
manual-gated (never CI). Tasks 1→2→3 are sequential; 4 depends on 3; 5–8 depend on 4 but are

### Task 1 — Model seam + value types (`propose_next`)

- **Goal:** the injectable model contract and the tiny value types the loop speaks, with a
  deterministic fake. Pure data + protocol; no I/O.
- **Prereqs:** none.
- **Files:**
  - `eval/minting_driver/__init__.py`
  - `eval/minting_driver/model.py` — `ToolCall(name: str, arguments: dict)`, `Done(reason: str)`,
    and a `Model` protocol: `propose_next(messages: list[Message]) -> ToolCall | Done`. `Message`
    is a minimal dict/dataclass carrying role + content + optional tool result.
  - `eval/minting_driver/fakes.py` — `ScriptedModel(steps: list[ToolCall | Done])`: pops one step
    per `propose_next` call; raises if called past the script (so a test can assert exact call
    count). This is the determinism anchor for Task 3.
  - `tests/test_minting_driver_model.py` — RED first: asserts `ScriptedModel` returns steps in
    order, that `Done` terminates a sequence, and that over-calling raises.
- **Validation:** `uv run pytest tests/test_minting_driver_model.py` green; no network.
- **Commit:** `eval(minting-driver): model seam + scripted fake (Task 1)`

### Task 2 — MCP stdio transport over the proxy subprocess

- **Goal:** a minimal interactive MCP client that spawns a given server command, sends one
  `\n`-delimited JSON-RPC request, and awaits its reply by `id`. Interactive (one at a time), NOT
  `converse` (which takes a pre-built frame list).
- **Prereqs:** Task 1.
- **Files:**
  - `eval/minting_driver/transport.py` — `class StdioMcp`: `__init__(command: list[str], env: dict)`
    spawns `subprocess.Popen(command, stdin=PIPE, stdout=PIPE, stderr=PIPE, env=...)`; `request(obj)
    -> reply` writes `json.dumps(obj).encode()+b"\n"`, flushes, and reads `\n`-delimited lines
    correlating by JSON-RPC `id`; `notify(obj)` sends without awaiting; `close()`. Mirror the
    send/await/correlate logic of `belay.replay.client._Replies`/`await_reply`
    (`replay/client.py:175-244,268-269`) — newline framing, `select`-bounded deadline, id match.
    **Decision (record in-file):** copy the ~30 lines rather than import `belay.replay`, so `eval/`
    is standalone and the isolation contract is unambiguous.
  - `eval/minting_driver/mcp.py` — thin helpers building the three request shapes: `initialize`,
    `tools/list`, `tools/call(name, arguments)`; and parsing a `tools/call` result into a
    tool-result `Message`.
  - `tests/test_minting_driver_transport.py` — RED first: a **fake MCP server** (a tiny stdlib
    script under `tests/fixtures/` or an in-test subprocess) that echoes canned replies; assert
    `request` correlates by id, awaits exactly one reply, and that interleaved notifications don't
    break correlation. Deterministic, no network, no LLM.
- **Validation:** `uv run pytest tests/test_minting_driver_transport.py` green; `import_guard` green.
- **Commit:** `eval(minting-driver): interactive MCP stdio transport (Task 2)`

### Task 3 — The sequential loop (THE shippable proof)

- **Goal:** the driver loop that ties model + transport together with **exactly one tool call in
  flight**, and the deterministic control-flow test that is this unit's acceptance criterion.
- **Prereqs:** Tasks 1, 2.
- **Files:**
  - `eval/minting_driver/loop.py` — `run_task(model: Model, transport, *, system, task,
    max_steps) -> Transcript`. Sequence: send `initialize` (await), `tools/list` (await), then loop:
    `model.propose_next(messages)` → if `Done` or step budget hit, stop; else send exactly one
    `tools/call`, **await its reply**, append the result to `messages`, repeat. Never issues a
    second call before the first's reply is in hand.
  - `tests/test_minting_driver_loop.py` — RED first, the **CI acceptance test**. Use `ScriptedModel`
    + a **fake transport** that records call order and, on each `tools/call`, asserts (via a
    re-entrancy/in-flight counter) that **no other call is outstanding** — never >1 in flight. Also
    assert: strict `initialize`→`tools/list`→call ordering; `Done` terminates; `max_steps` caps an
    endless model; each tool result is fed back before the next proposal. No network, no LLM.
- **Validation:** `uv run pytest tests/test_minting_driver_loop.py` green — this is the spec's
  "loop provably issues tool calls sequentially" criterion.
- **Commit:** `eval(minting-driver): sequential one-call-in-flight loop + control-flow test (Task 3)`

### Task 4 — Proxy/env wiring helper (gated path)

- **Goal:** build the proxy command + the **gated** env (all three vars) so captured turns are
  snapshotted and verifiable — the resolved decision #1.
- **Prereqs:** Task 3.
- **Files:**
  - `eval/minting_driver/capture.py` — `proxy_command(server_command: list[str]) ->
    list[str]` returns `[sys.executable, "-m", "belay.proxy", *server_command]` (NB: the entire
    argv after `-m belay.proxy` IS the server command — **no `--` separator**, `proxy.py:475-480`).
    `gated_env(*, trace_dir, scope, snapshot_dir, base=os.environ) -> dict` sets `BELAY_TRACE_DIR`,
    `BELAY_SANDBOX_SCOPE`, `BELAY_SNAPSHOT_DIR`; **raises** if scope is set without snapshot_dir
    (the proxy hard-refuses, `proxy.py:544-554`) so the driver fails fast, not the subprocess.
  - `tests/test_minting_driver_capture.py` — RED first: assert `proxy_command` shape (no `--`),
    assert `gated_env` sets all three and raises on the scope-without-snapshot mistake. Pure, no
    subprocess spawn — deterministic, CI.
- **Validation:** `uv run pytest tests/test_minting_driver_capture.py` green.
- **Commit:** `eval(minting-driver): gated proxy/env wiring helper (Task 4)`

## 5. Environment & determinism

- **Env vars:** `BELAY_TRACE_DIR`, `BELAY_SANDBOX_SCOPE`, `BELAY_SNAPSHOT_DIR` (gated capture);
  `ANTHROPIC_API_KEY` (Anthropic client); `OPENAI_BASE_URL`/`OPENAI_API_KEY` (local client);
  `BELAY_EVAL_LIVE` (smoke opt-in).
- **External tools:** Node 20/22; `npx -y @modelcontextprotocol/server-filesystem@<pin>`,
  `npx -y mcp-server-commands@<pin>`. Pinned for reproducible mints.
- **Determinism requirement:** no CI test may depend on clock/network/random/model. The loop is
  deterministic given `ScriptedModel`; anything non-deterministic is behind the manual gate.

## 6. Edge cases & error handling (and the verdict each implies)

- **Capture-only (no snapshot) turn** → `state_handle: absent` → `verify_turn` = **UNVERIFIED**,
  instance = `no-verifiable-turns`. The driver avoids this by always using the gated env (Task 4);
  the smoke asserts a verifiable turn so a regression to capture-only fails loudly.
- **Scope set without snapshot dir** → proxy hard-refuses (exit 2). `gated_env` raises first, in
  the driver, with a clear message — never a mystery subprocess crash.
- **Model never emits `Done`** → `max_steps` caps the loop; the transcript records truncation. Not
  an error, a bounded stop.
- **Tool reply never arrives / server exits** → transport surfaces it as an explicit failure
  outcome and stops the loop (mirrors `converse`'s SERVER_EXITED); never a silent hang, never a
  second call issued.
- **Non-darwin / no model key / SDK absent** → the smoke and real clients **skip** (guarded); they
  never fabricate a pass. The core loop test is unaffected.
- **`run_batch` sees zero verifiable turns in a mint** → that is the runner's `INSTRUMENT SUSPECT`
  path (already built); the driver's job is to make that outcome mean "setup is wrong," which the
  smoke check catches pre-scale (R6 "verify one before scaling").
